# Product

## Register

product

## Users

Delano is used by maintainers, operators, and coding agents working inside repositories that use file-backed delivery contracts. Maintainers evolve the handbook, runtime, installer, and viewer. Operators use Delano locally to inspect project state, validate contracts, and navigate delivery artifacts. Coding agents use the same contracts and adapter notes to work without depending on a single vendor-specific workflow.

The primary interface is a localhost-only document reader and navigator for `.project` contracts, supported by a thin CLI and runtime scripts.

## Product Purpose

Delano turns delivery state into readable, inspectable local contracts. It exists to keep planning, execution, validation, and evidence traceable on disk while allowing different coding agents and operators to follow the same operating model.

Success means a user can open a repository, understand current delivery state, navigate specs, plans, workstreams, tasks, and evidence, and validate the runtime without guessing where truth lives.

## Brand Personality

Quiet, document-like, deliberate.

The product should feel calm, precise, and operational. It should privilege careful reading, clear hierarchy, and trustworthy status cues over visual novelty.

## Anti-references

Delano should not feel flashy, decorative, or marketing-led. Avoid generic SaaS dashboard tropes, ornamental UI, playful mascot-driven agent tooling, heavy enterprise PM-suite density, dark cyber-terminal styling, and promotional landing-page composition inside the product surface.

## Design Principles

1. Treat contracts as the product surface.
2. Make document structure easier to inspect, not louder.
3. Preserve operator trust through quiet status, clear provenance, and predictable navigation.
4. Keep the viewer read-only in spirit and presentation.
5. Prefer deliberate hierarchy over decoration.

## Accessibility & Inclusion

No additional accessibility requirements beyond the project baseline were requested.
