# Delano Guarded Viewer

Local shadcn-built frontend for browsing and reviewing `.project` markdown contracts.

- Review mode: serves files from `.project`, stores annotations in `.project/viewer/annotations.json`, and writes canonical markdown only through explicit preview/apply endpoints.
- Default starting URL: `http://127.0.0.1:3977`
- Override starting port: `DELANO_VIEWER_PORT=3987 npm run viewer`
- Multiple viewers can run at once. If the starting port is already in use, the viewer tries the next available port and prints the actual URL.

Run from the repository root:

```bash
npm run viewer
```

The viewer indexes `.project/context/**/*.md`, `.project/templates/**/*.md`, and `.project/projects/**/*.md`. It derives artifact roles (`spec`, `plan`, `workstream`, `task`, `research`, `progress`, `decision`, `context`, `template`), status fields, task/workstream relationships, relationship-like wikilinks, snippets, and renders markdown in a Tolaria-inspired review pane layout. Every Markdown file below a project's `research/` subtree stays research evidence, including nested files named `progress.md`.

The `/api/index` response also exposes `contextPack` metadata from the shared context reader. It includes ordered context file metadata, missing-file warnings, and profile command references such as `delano context read --profile implementation`. It does not embed full context file content.

Repository and worktree context is server-authoritative. Searchable controls at the top of the sidebar select only registered repositories and fresh Git-reported worktrees; their details control exposes the full paths, branch or detached state, HEAD, `.project` source, availability, and divergence. The browser stores a versioned pair of opaque repository/worktree IDs and restores it only when the fresh inventory still reports that selection as available. `GET /api/context` returns that inventory, selected provenance, divergence/dirty state, and primary-only write capability. `POST /api/context` accepts only IDs from that inventory, tears down root-scoped watchers/caches/SSE clients, and returns the rebuilt index. The same context and schema-derived artifact enum options are included in `/api/index`; missing or malformed schemas produce `schemaOptionsError` rather than a fallback vocabulary. Linked worktrees are inspect-only: edit and annotation/handover controls are disabled with an explanation, and the server independently rejects annotation, apply, and handover writes with `403`.

Project folders get a right-side outline for the spec, plan, decisions, workstreams, and tasks, with one Research entry opening every nested research document and one Progress entry opening that project's complete update list. Project overview is a current-state delivery dashboard rather than a file inventory: it shows a flat project summary, task-state execution map, canonical spec brief, workstream completion and active-task drilldowns, and the latest progress evidence. The dashboard derives only from indexed contracts; it does not fabricate historical trends or forecasts. Project selectors and repository/worktree changes preserve the exact current route when it remains valid, otherwise they translate it to the closest equivalent project section or document role. Project names in the Workspace Projects table are explicit navigation links: they select that project and open Project overview. Workspace navigation is ordered Projects, Tasks, Context pack, Annotations, Warnings, and Blockers. The Tasks badge counts planned, ready, in-progress, and blocked tasks, while the Tasks view still includes every indexed status (including done and deferred) with project, workstream, priority, estimate, and update context. The Annotations badge counts open comments, while its page retains non-deleted history. Task Project and Workstream filters are searchable multi-selects built from indexed relationships. Status, Priority, and Estimate options come from the canonical task schema exposed by `/api/index`; display labels never become filter values, and a schema loading error is shown instead of replaced by UI constants. Viewer data tables use one ledger boundary, comfortable shared row padding, sortable headers, searchable option or text filters, clear-one/clear-all behavior, filtered pagination totals, content-aware column widths, and top-aligned rows. Tables with Updated data open newest-first except Context pack, whose canonical reader order is preserved. Updated and Created columns provide Today, Last 7 days, Last 30 days, This year, and custom inclusive date-range filters. Workspace Projects also shows each contract's Created timestamp. Opening a workstream shows its indexed tasks below the contract body with canonical status, dependency-aware availability, priority/estimate context, and direct task navigation.

Annotation count semantics are intentionally asymmetric: the sidebar badge counts only open comments, while the table keeps every non-deleted annotation, including resolved and closed history. Its Status column uses a selectable filter over the statuses present in that history.

Task readers surface the useful contract context above the task body: a direct link to the parent workstream, status, priority, estimate, acceptance-criteria progress and IDs, plus non-empty execution and traceability metadata such as story, operating mode, dependency, conflict, owner, and tracker references. The acceptance summary links to the full checklist in the task contract.

The reader starts in reading mode, where text selection stays native and never creates a yellow mark, composer, or annotation write. Annotation highlights and selected-text creation are enabled only after the user explicitly activates **Review**. Existing annotations do not open Review automatically. The fixed review panel overlays the header and document without changing the reading pane width or line length. Writable primary worktrees support sticky create/edit popovers (unsaved feedback closes only through Save, the close button, Escape, or leaving Review); linked worktrees can inspect existing highlights and read-only popovers but cannot create or change annotations. The panel provides deterministic markdown/JSON export. The primary review output is agent handover: `POST /api/handover` writes a handover file under `.project/viewer/handovers/` with the selected annotations and supports ChatGPT, Codex, Claude Code, Claude, and T3 Code. ChatGPT is the default and opens through its registered `codex://new` desktop link. Claude Code and Claude use their prompt-bearing desktop links, Codex uses terminal launch, and every destination provides a copyable handover fallback. T3 Code uses the installed `t3code` CLI to resolve this repository, create its T3 project when policy permits, create a fresh thread, submit the handover prompt, and reveal the T3 app. The same endpoint supports work dispatch on tasks and workstreams via `intent: "start" | "review"`: start hands the agent the contract to implement, review asks it to verify acceptance criteria and evidence (including captured annotations when present); these intents reference the contract directly and only write a handover file when annotation feedback exists. The receiving agent works under its own permissions; the viewer never writes canonical markdown through handover. File application remains guarded: the server checks repo-relative `.project` paths, current file hashes, and explicit confirmation before writing canonical markdown.

The reader also supports in-place editing. Pressing `E` (or Ctrl/Cmd+E, or the Edit button) switches the document body into a TipTap WYSIWYG editor with markdown-true serialization (headings, lists, task lists, tables, code fences, links). Frontmatter is never editable: it renders as a locked properties card and is reattached byte-identical on save. Saves go through the same guarded `POST /api/apply` path (`expectedHash` + `confirm`), so a stale baseline returns 409 and surfaces a conflict banner with explicit reload-theirs or keep-mine choices — the viewer never overwrites external changes silently. Documents whose markdown the serializer would normalize (for example table padding) show a "formatting will be tidied on save" hint instead of blocking.

The viewer watches `.project` recursively and streams changes over `GET /api/events` (SSE: `index-changed`, `doc-changed`, heartbeat; debounced rescan because Windows coalesces bulk file events). Open documents refresh live within ~2 seconds of an external write, changed blocks flash briefly, and the workspace index refreshes silently. A topbar Activity panel (backed by `GET /api/activity`, an in-memory ring buffer capped at 200 events) lists recent file changes newest-first with a pulsing indicator while events arrive — this is how you watch an agent work after a handover. Dispatching a handover leaves a persistent "handed over" banner on the document that links straight to the activity panel. If an open editor's file changes on disk, a clean editor follows silently and a dirty editor gets the conflict banner instead of losing work.

The browser client lives in `.delano/viewer/ui` and is built with the shadcn CLI plus real shadcn/Radix components. Run `npm --prefix .delano/viewer/ui run build` after changing the UI, then `npm run build:assets` before packaging.

Convenience buttons can open the selected markdown file's containing folder in the system explorer or open the file in VS Code. These actions are guarded so they only target markdown files inside `.project`.
