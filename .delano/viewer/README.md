# Delano Guarded Viewer

Local shadcn-built frontend for browsing and reviewing `.project` markdown contracts.

- Review mode: serves files from the selected worktree's `.project`, keeps draft findings in browser-local storage, publishes explicit per-session reviews under `.project/reviews/`, and writes canonical markdown only through guarded preview/apply endpoints.
- Default starting URL: `http://127.0.0.1:3977`
- Override starting port: `DELANO_VIEWER_PORT=3987 npm run viewer`
- Multiple viewers can run at once. If the starting port is already in use, the viewer tries the next available port and prints the actual URL.

Run from the repository root:

```bash
npm run viewer
```

The viewer indexes `.project/context/**/*.md`, `.project/roadmap/**/*.md`, `.project/templates/**/*.md`, and `.project/projects/**/*.md`. It derives artifact roles (`spec`, `plan`, `roadmap-item`, `workstream`, `task`, `research`, `progress`, `decision`, `context`, `template`), status fields, task/workstream relationships, relationship-like wikilinks, snippets, and renders markdown in a Tolaria-inspired review pane layout. Every Markdown file below a project's `research/` subtree stays research evidence, including nested files named `progress.md`.

The `/api/index` response also exposes `contextPack` metadata from the shared context reader. It includes ordered context file metadata, missing-file warnings, and profile command references such as `delano context read --profile implementation`. It does not embed full context file content.

Repository and worktree context is server-authoritative. Searchable controls at the top of the sidebar select only registered repositories and fresh Git-reported worktrees; their details expose full local paths, branch or detached state, HEAD, `.project` source, availability, divergence, and dirty provenance. The browser stores only a versioned pair of opaque repository/worktree IDs and restores it when the fresh inventory still validates the selection. `GET /api/context` returns that inventory plus selected-context risk and explicit `dispatch`, `review`, `publishReview`, and `applyContract` capabilities. Primary/linked role is provenance, not authorization: any fresh registered selected worktree receives the same guarded capabilities. Every launch or write revalidates context identity, containment, and the relevant hash, and stale, switched, or deleted worktrees fail without fallback. `POST /api/context` accepts only IDs from the inventory, tears down root-scoped watchers/caches/SSE clients, and returns the rebuilt index.

Project folders get a right-side outline for the spec, plan, decisions, workstreams, and tasks, with one Research entry opening every nested research document and one Progress entry opening that project's complete update list. Project overview is a current-state delivery dashboard rather than a file inventory: it shows a flat project summary, task-state execution map, canonical spec brief, workstream completion and active-task drilldowns, and the latest progress evidence. The dashboard derives only from indexed contracts; it does not fabricate historical trends or forecasts. Project selectors and repository/worktree changes preserve the exact current route when it remains valid, otherwise they translate it to the closest equivalent project section or document role. Project names in the Workspace Projects table are explicit navigation links: they select that project and open Project overview. Workspace navigation includes Projects, Tasks, Context pack, Reviews, legacy Annotations, Warnings, and Blockers. Reviews lists tracked `.project/reviews/` artifacts; Annotations remains a compatibility view for legacy state and explicit migration. The Tasks badge counts planned, ready, in-progress, and blocked tasks, while the Tasks view still includes every indexed status (including done and deferred) with project, workstream, priority, estimate, and update context. Task Project and Workstream filters are searchable multi-selects built from indexed relationships. Status, Priority, and Estimate options come from the canonical task schema exposed by `/api/index`; display labels never become filter values, and a schema loading error is shown instead of replaced by UI constants. Viewer data tables use one ledger boundary, comfortable shared row padding, sortable headers, searchable option or text filters, clear-one/clear-all behavior, filtered pagination totals, content-aware column widths, and top-aligned rows. Tables with Updated data open newest-first except Context pack, whose canonical reader order is preserved. Updated and Created columns provide Today, Last 7 days, Last 30 days, This year, and custom inclusive date-range filters. Workspace Projects also shows each contract's Created timestamp. Opening a workstream shows its indexed tasks below the contract body with canonical status, dependency-aware availability, priority/estimate context, and direct task navigation.

When roadmap capability is present, Workspace navigation adds a Roadmap board with `now | next | later` lanes and an explicit terminal archive. Cards derive linked-project states, task totals, latest canonical activity, source links, and advisory staleness from `.project`; no reverse links are persisted. Keyboard-operable move and promotion actions require a current item hash, whitelisted fields, capability, and explicit confirmation, return `409` without writing on conflict, and may offer the existing `start` handover only as a separate optional action after promotion succeeds. Item or linked-project SSE changes refresh the projection and give only affected cards a bounded highlight plus textual Updated feedback.

Legacy annotation count semantics remain available only for migration compatibility. Published review counts and lifecycle derive from review schema status and finding status.

Task readers surface the useful contract context above the task body: a direct link to the parent workstream, status, priority, estimate, acceptance-criteria progress and IDs, plus non-empty execution and traceability metadata such as story, operating mode, dependency, conflict, owner, and tracker references. The acceptance summary links to the full checklist in the task contract.

The reader starts in reading mode, where selection remains native. Review mode saves draft findings in browser-local storage scoped by repository, worktree, and source path. Publishing requires explicit confirmation and writes one schema-valid Markdown artifact under `.project/reviews/`; uncommitted source requires a visible confirmation and records a null source commit while retaining the normalized content hash. Publication never commits, pushes, or posts remotely. Published reviews are indexed with lifecycle states (`open`, `resolved`, `archived`), stay current when unrelated HEAD movement leaves the reviewed content unchanged, and become stale when the normalized source hash changes. Review handovers reference the tracked review path directly; task/workstream start and review handovers reference their contracts directly and launch the agent in the selected worktree. Legacy `.project/viewer/annotations.json` and generated handovers are read only for explicit, idempotent, non-destructive migration. File application remains guarded by selected-context capability, containment, expected hash, and explicit confirmation.

The reader also supports in-place editing. Pressing `E` (or Ctrl/Cmd+E, or the Edit button) switches the document body into a TipTap WYSIWYG editor with markdown-true serialization (headings, lists, task lists, tables, code fences, links). Frontmatter is never editable: it renders as a locked properties card and is reattached byte-identical on save. Saves go through the same guarded `POST /api/apply` path (`expectedHash` + `confirm`), so a stale baseline returns 409 and surfaces a conflict banner with explicit reload-theirs or keep-mine choices — the viewer never overwrites external changes silently. Documents whose markdown the serializer would normalize (for example table padding) show a "formatting will be tidied on save" hint instead of blocking.

The viewer watches `.project` recursively and streams changes over `GET /api/events` (SSE: `index-changed`, `doc-changed`, heartbeat; debounced rescan because Windows coalesces bulk file events). Open documents refresh live within ~2 seconds of an external write, changed blocks flash briefly, and the workspace index refreshes silently. A topbar Activity panel (backed by `GET /api/activity`, an in-memory ring buffer capped at 200 events) lists recent file changes newest-first with a pulsing indicator while events arrive — this is how you watch an agent work after a handover. Dispatching a handover leaves a persistent "handed over" banner on the document that links straight to the activity panel. If an open editor's file changes on disk, a clean editor follows silently and a dirty editor gets the conflict banner instead of losing work.

The browser client lives in `.delano/viewer/ui` and is built with the shadcn CLI plus real shadcn/Radix components. Run `npm --prefix .delano/viewer/ui run build` after changing the UI, then `npm run build:assets` before packaging.

The npm package owns this executable Viewer runtime. `delano install` does not copy it into target repositories; any existing repository-local `.delano/viewer` tree is inert legacy data and is never removed automatically.

Convenience buttons can open the selected markdown file's containing folder in the system explorer or open the file in VS Code. These actions are guarded so they only target markdown files inside `.project`.
